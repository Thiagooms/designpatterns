Desafio — Case 1

Padrão Adapter

Contexto

Seu sistema de pagamentos usa uma interface própria chamada ProcessadorPagamento com o método pagar(double valor). Todo o código do sistema depende dessa interface.

A empresa contratou um gateway externo — a biblioteca PagSeguroLegado — que já existe pronta e não pode ser modificada. O problema: ela expõe um método com uma assinatura completamente diferente: efetuarCobranca(String moeda, int centavos).

O desenvolvedor tentou resolver o problema da forma mais rápida possível e o resultado está abaixo. O código até funciona, mas introduz um smell code sério que vai se repetir cada vez que um novo gateway for integrado.

Sobre o smell code

Shotgun Surgery + Violated Encapsulation

O Shotgun Surgery aparece quando uma mudança exige alterações espalhadas em vários lugares do código. Aqui, a lógica de conversão (valor * 100, moeda "BRL") está duplicada direto no cliente — se o gateway mudar a forma de cobrar, você precisa encontrar e alterar todos os pontos do sistema que o chamam.

O Violated Encapsulation complementa o problema: o cliente (ServicoVenda) precisa conhecer detalhes internos do gateway externo — como ele representa centavos, qual string de moeda usar — sendo que isso não é responsabilidade dele.

O sintoma direto é a conversão (int)(valor * 100) e a string "BRL" espalhadas pelo código do cliente, em vez de estar encapsuladas em um único lugar.

Código atual

smell code 
// Biblioteca externa — não pode ser modificada
public class PagSeguroLegado {

    public void efetuarCobranca(String moeda, int centavos) {
        System.out.println("[PagSeguro] Cobrando " + centavos + " centavos em " + moeda);
    }
}
// Interface que o sistema espera
public interface ProcessadorPagamento {
    void pagar(double valor);
}
// Cliente que usa o gateway diretamente — sem nenhum adaptador
public class ServicoVenda {

    private PagSeguroLegado gateway = new PagSeguroLegado();

    public void finalizarCompra(double valor) {
        System.out.println("Finalizando compra de R$ " + valor);

        // Lógica de conversão vazando para o cliente
        int centavos = (int) (valor * 100);
        gateway.efetuarCobranca("BRL", centavos);
    }
}
public class Main {

    public static void main(String[] args) {
        ServicoVenda venda = new ServicoVenda();
        venda.finalizarCompra(149.90);
        venda.finalizarCompra(39.00);
    }
}
Um segundo gateway — MercadoPagoLegado — precisa ser integrado na próxima sprint. Ele também tem uma assinatura diferente: realizarDebito(int valorEmCentavos, String codigoMoeda). O que acontece com o ServicoVenda se você continuar sem o padrão Adapter?

O que você deve implementar

  1. Crie uma classe PagSeguroAdapter que implemente ProcessadorPagamento e delegue para PagSeguroLegado internamente.
  2. Remova a lógica de conversão de centavos e a string "BRL" do ServicoVenda — ela pertence ao adapter.
  3. Faça o ServicoVenda depender apenas de ProcessadorPagamento, sem saber nada sobre o gateway concreto.
  4. Garanta que a saída do programa continue idêntica após a refatoração.

A saída do programa deve continuar sendo a mesma após a refatoração.
