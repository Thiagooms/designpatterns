Desafio — Case 4

Smell Code para refatorar

Contexto

Um e-commerce processa pagamentos aplicando descontos antes de cobrar o cliente. O desconto pode ser: VIP (15% sobre o valor), Cupom (R$20,00 fixo) ou nenhum. O pagamento pode ser feito via Cartão de Crédito, PIX ou Boleto.

O time comercial quer adicionar o desconto Cashback (5% devolvido) e o método de pagamento Cartão de Débito na próxima versão.

Sobre o smell code

Divergent Change + Violação do SRP

O ProcessadorDePagamento muda por dois motivos completamente diferentes: se uma nova regra de desconto surgir, a classe muda; se um novo método de pagamento for suportado, a classe também muda.

As duas dimensões são totalmente independentes — qualquer desconto deve poder se combinar com qualquer método de pagamento — mas estão acopladas dentro do mesmo método.

Código atual

smell code 
public class ProcessadorDePagamento {

    public void processar(double valor, String desconto, String metodoPagamento) {

        double valorFinal = valor;

        // Motivo de mudança 1: Cálculo de desconto
        if (desconto.equals("VIP")) {
            valorFinal = valor * 0.85;
            System.out.println("Desconto VIP de 15% aplicado. Novo valor: R$" + valorFinal);

        } else if (desconto.equals("CUPOM")) {
            valorFinal = valor - 20.0;
            System.out.println("Cupom de R$20,00 aplicado. Novo valor: R$" + valorFinal);

        } else if (desconto.equals("NENHUM")) {
            System.out.println("Sem desconto. Valor: R$" + valorFinal);

        } else {
            throw new IllegalArgumentException("Tipo de desconto desconhecido.");
        }

        // Motivo de mudança 2: Método de pagamento
        if (metodoPagamento.equals("CREDITO")) {
            System.out.println("Processando pagamento no cartão de crédito...");
            System.out.println("Cobrança de R$" + valorFinal + " aprovada.");

        } else if (metodoPagamento.equals("PIX")) {
            System.out.println("Gerando chave PIX...");
            System.out.println("Transferência de R$" + valorFinal + " confirmada.");

        } else if (metodoPagamento.equals("BOLETO")) {
            System.out.println("Gerando boleto bancário...");
            System.out.println("Boleto de R$" + valorFinal + " emitido com vencimento em 3 dias.");

        } else {
            throw new IllegalArgumentException("Método de pagamento desconhecido.");
        }
    }
}
Desconto e método de pagamento variam de forma completamente independente: VIP + PIX, CUPOM + BOLETO, NENHUM + CREDITO — qualquer combinação deve ser possível. Como separar essas responsabilidades de forma que adicionar Cashback ou Cartão de Débito não exija tocar em nenhuma classe já existente?

A solução deve usar duas interfaces de estratégia independentes: uma para cálculo de desconto e outra para método de pagamento. O ProcessadorDePagamento deve receber ambas pelo construtor. Atenção: o resultado da estratégia de desconto (double) é a entrada da estratégia de pagamento — o fluxo entre as duas não é opcional.
